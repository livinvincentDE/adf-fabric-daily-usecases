Incremental Data Load
Load only new or modified records using watermark columns to avoid full-table scans on every run
🔁
Incremental Data Load
Load only new or modified records using watermark columns to avoid full-table scans on every run
Prerequisites
Azure Data Factory instance
Source: Azure SQL with a ModifiedDate / LastUpdated column
Destination: Azure SQL or ADLS Gen2
Control table in Azure SQL to store watermarks
Steps
1
Create a watermark control table
CREATE TABLE dbo.watermark_store (TableName NVARCHAR(100), WatermarkValue DATETIME2); INSERT INTO dbo.watermark_store VALUES ('orders', '1900-01-01');
2
Add Lookup activity — get last watermark
Lookup → Query: SELECT WatermarkValue FROM dbo.watermark_store WHERE TableName = 'orders'. Feeds lower bound to Copy activity.
3
Add Copy Data with dynamic query
Copy → Source → Query: @concat('SELECT * FROM dbo.orders WHERE ModifiedDate > ''', activity('LookupOldWatermark').output.firstRow.WatermarkValue, '''')
4
Second Lookup — get new max watermark
After Copy: Lookup → SELECT MAX(ModifiedDate) AS NewWatermark FROM dbo.orders
5
Update watermark with Stored Procedure
SP activity → sp_update_watermark with param @NewWatermark = @{activity('LookupNewWatermark').output.firstRow.NewWatermark}
6
Schedule with Tumbling Window Trigger
Trigger → New → Tumbling Window → recurrence: 1 hour. Guarantees no overlapping windows and safe backfill.
💡
Pro Tip
Always update the watermark AFTER a successful copy. If the copy fails, the watermark stays at the previous checkpoint so the next run reprocesses safely.

Metadata-Driven Pipeline
Use a control table to drive dynamic multi-table ingestion — one pipeline handles hundreds of tables without code duplication
🧩
Metadata-Driven Pipeline
Use a control table to drive dynamic multi-table ingestion — one pipeline handles hundreds of tables without code duplication
Prerequisites
Azure Data Factory instance
Azure SQL control table listing source tables and load configs
Parameterised linked datasets
Steps
1
Design the pipeline control table
CREATE TABLE dbo.pipeline_control (Id INT IDENTITY PRIMARY KEY, SourceSchema NVARCHAR(50), SourceTable NVARCHAR(100), DestContainer NVARCHAR(100), LoadType NVARCHAR(20), IsActive BIT);
2
Add Lookup to read active configs
Lookup → SELECT * FROM dbo.pipeline_control WHERE IsActive = 1. Disable 'First row only'.
3
Add ForEach over control rows
ForEach → Items: @activity('LookupConfig').output.value. Batch Count: 5–10. Sequential = false.
4
Add parameterised Copy inside ForEach
Copy → Source param: tableName = @item().SourceTable. Sink folder: @concat(item().DestFolder, '/', formatDateTime(utcNow(), 'yyyy/MM/dd'))
5
Handle full vs incremental load types
If Condition → @equals(item().LoadType, 'incremental') → True: Copy with WHERE filter. False: full table copy.
6
Log run results per table
Script activity → INSERT into run_log with table name, rows copied, timestamp, status.
💡
Pro Tip
Store the control table in Azure SQL, not a JSON file. SQL rows can be activated or deactivated at runtime without touching pipeline code.

🔄
ForEach Loop Pipeline
Process multiple files or tables dynamically by iterating over a list with ForEach, enabling parallel execution
Prerequisites
Azure Data Factory instance
ADLS Gen2 source folder with multiple files
Azure SQL or ADLS Gen2 destination
Steps
1
Add Get Metadata to list files
Get Metadata → Dataset: ADLS Gen2 folder → Field list: 'childItems'. Returns array of file objects.
2
Filter to target file type
Filter → Items: @activity('GetMetadata').output.childItems → Condition: @endswith(item().name, '.csv')
3
Add ForEach over filtered files
ForEach → Items: @activity('FilterCSVFiles').output.value. Sequential = false, Batch Count = 10.
4
Configure Copy inside ForEach
Copy → Source: @concat('landing/', item().name). Sink: Azure SQL or partitioned ADLS folder.
5
Log each file result
Script activity → INSERT into load_log with file name, rows copied, timestamp.
6
Handle empty folder gracefully
If Condition before ForEach → @greater(length(activity('FilterCSVFiles').output.value), 0) → True: ForEach. False: Set Variable 'No files found'.
💡
Pro Tip
Set ForEach Batch Count to 10–15 for typical file volumes. Exceeding 20 risks throttling. Monitor 'Queue duration' in activity run details.

Retry & Error Handling
Configure retry policies, failure branches, alert webhooks, and structured error logging for robust pipeline operations
Prerequisites
Existing ADF pipeline
Azure SQL error log table
Teams webhook URL (optional)
Steps
1
Set retry on individual activities
Activity → Settings → Retry: 3. Interval: 30 seconds. ADF auto-retries transient failures before marking as failed.
2
Wire up a failure dependency branch
Drag from the red (Failure) port of Copy activity to a new activity. Executes only on failure — ADF's catch block.
3
Capture error message in variable
Failure branch → Set Variable → ErrorMsg → @{activity('CopyData').error.message}
4
Send Teams alert
Web activity → POST → URL: Teams webhook → Body: {"text": "❌ Pipeline @{pipeline().Pipeline} failed: @{variables('ErrorMsg')}"}
5
Log error to Azure SQL
Script activity → INSERT INTO dbo.pipeline_errors (PipelineName, ErrorMessage, FailedAt) VALUES (..., GETDATE())
6
Set pipeline timeout
Pipeline Settings → Timeout: 2 hours. Add cleanup on grey 'On Completion' port — runs whether pipeline succeeded or failed.
💡
Pro Tip
Use 'On Completion' (grey) for cleanup, 'On Failure' (red) for alerts, 'On Success' (green) for next steps. All three can coexist on the same source activity.

🔁
Incremental Data Processing
Process only new or updated data using Delta MERGE and watermark strategy
Prerequisites
Source Delta table with a timestamp column
Control table for watermarks
Steps
1
Create watermark control table
spark.sql("CREATE TABLE IF NOT EXISTS watermark_ctrl (table_name STRING, last_load_ts TIMESTAMP)")
2
Read current watermark
wm = spark.sql("SELECT last_load_ts FROM watermark_ctrl WHERE table_name='orders'").collect()[0][0]
3
Filter source to new rows only
new_df = spark.read.table("bronze_orders").filter(col("updated_at") > wm)
4
Upsert to Silver using Delta MERGE
DeltaTable.forName(spark,"silver_orders").alias("t").merge(new_df.alias("s"), "t.order_id = s.order_id").whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()
5
Update watermark
new_wm = new_df.agg({"updated_at":"max"}).collect()[0][0]; spark.sql(f"UPDATE watermark_ctrl SET last_load_ts='{new_wm}' WHERE table_name='orders'")
💡
Pro Tip
Always update the watermark AFTER a successful merge, never before, to prevent data loss on failure.

ADF Expression — Extract date from filename ADF Expression
-- filename: customer_2025-07-10.csv -- substring(item().name, 9, 10) → "2025-07-10" @formatDateTime(substring(item().name, 9, 10), 'yyyy-MM-dd') -- Then stamp the staging table: UPDATE stg_customer SET file_date = '@{variables(''var_file_date'')}';
sp_upsert_dim_customer_scd1 — Full Stored Procedure SCD Type 1 · UPSERT
CREATE OR ALTER PROCEDURE sp_upsert_dim_customer_scd1 AS BEGIN SET NOCOUNT ON; -- ① UPDATE: existing customers with newer data and any changed attribute UPDATE tgt SET name = src.name, email = src.email, address = src.address, last_updated_date = src.file_date FROM dim_customer tgt INNER JOIN stg_customer src ON tgt.customer_id = src.customer_id WHERE TRY_CAST(src.file_date AS DATE) > TRY_CAST(tgt.last_updated_date AS DATE) AND ( tgt.name <> src.name OR tgt.email <> src.email OR tgt.address <> src.address ); -- ② INSERT: new customer_ids that don't exist in dim yet INSERT INTO dim_customer (customer_id, name, email, address, last_updated_date) SELECT customer_id, name, email, address, file_date FROM stg_customer src WHERE NOT EXISTS ( SELECT 1 FROM dim_customer tgt WHERE tgt.customer_id = src.customer_id ); END;

💡
Key Design Decisions
Why TRY_CAST? The file_date and last_updated_date are stored as VARCHAR — TRY_CAST(... AS DATE) handles the comparison safely without failing on malformed values.

Why check attribute changes? The WHERE clause prevents unnecessary UPDATE I/Os when a newer file re-sends the same customer with no changes — a common pattern in daily full-extract loads.

Why TRUNCATE stg_customer per run? Staging should only contain one day's data. If you append, day 10 records would incorrectly compete with day 12 records in the merge logic.

SCD Type 2 — Preserve Full History
Slowly Changing Dimension Type 2 keeps every version of a record. When an attribute changes, the old row is expired (end-dated) and a new row is inserted with a fresh surrogate key. This guide uses the same customer_2025-07-10/11/12.csv files but now against a dim_customer_scd2 table with surrogate keys, effective_date, expiry_date, and is_current flags — implemented in both T-SQL (Fabric Warehouse) and PySpark Delta MERGE (Fabric Lakehouse).
⚖️ SCD Type 1 vs Type 2 — Side by Side
🔄 SCD Type 1
✦ Overwrites old value in place

✦ Simple UPSERT (UPDATE + INSERT)
✦ No surrogate key needed
✦ Smaller table footprint

📜 SCD Type 2

✦ Requires surrogate key (sk_customer)

✦ Larger table, richer analytics
✦ Use when: history matters (e.g. address changes affect revenue attribution)

🗂️ Step-by-Step Setup
01
Create dim_customer_scd2 with surrogate key
The table has sk_customer (auto-increment surrogate), the natural key customer_id, tracked attributes, plus effective_date, expiry_date, and is_current. The sentinel value 9999-12-31 marks open-ended (still-active) records.
02
Load staging table from daily CSV (same as SCD1)
ADF Copy Activity truncates and reloads stg_customer daily. Use the same ADF expression to extract file_date from the filename. The staging schema is identical to SCD Type 1.
03
EXPIRE changed rows (UPDATE existing current rows)
For rows where the natural key matches AND any attribute has changed: set is_current = 0 and expiry_date = file_date - 1 day. The old record is now a historical record — never deleted.
04
INSERT new version rows for changed + new customers
Insert all rows from staging where: (a) the customer is brand new to dim, OR (b) the customer's row was just expired in step 03. New rows get effective_date = file_date, expiry_date = '9999-12-31', is_current = 1.
05
Validate: query current vs historical records
After all 3 days: SELECT * FROM dim_customer_scd2 WHERE is_current = 1 returns 5 active records. WHERE is_current = 0 returns 2 expired records (old John Doe, old Jane Smith).
06
PySpark alternative: Delta MERGE for Lakehouse SCD2
In Fabric Lakehouse, use Delta's MERGE statement with whenMatchedUpdate to expire and whenNotMatchedInsert to add new versions. Full PySpark code shown below.
💾 SQL Objects — T-SQL (Fabric Warehouse)
dim_customer_scd2 — Table DDL Fabric Warehouse · T-SQL
CREATE TABLE dim_customer_scd2 ( sk_customer INT IDENTITY(1001,1), -- surrogate key, auto-increment customer_id INT NOT NULL, -- natural/business key name VARCHAR(100), email VARCHAR(100), address VARCHAR(200), effective_date DATE NOT NULL, -- when this version became active expiry_date DATE NOT NULL -- 9999-12-31 = still current DEFAULT '9999-12-31', is_current BIT NOT NULL -- 1 = active record, 0 = expired DEFAULT 1, dw_created_date DATETIME2 DEFAULT GETDATE() -- audit: when row was written to DW );
sp_scd2_dim_customer — Full Stored Procedure SCD Type 2 · EXPIRE + INSERT
CREATE OR ALTER PROCEDURE sp_scd2_dim_customer AS BEGIN SET NOCOUNT ON; -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ -- STEP 1: EXPIRE rows where customer exists AND any -- tracked attribute has changed -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ UPDATE tgt SET is_current = 0, expiry_date = DATEADD(DAY, -1, TRY_CAST(src.file_date AS DATE)) FROM dim_customer_scd2 tgt INNER JOIN stg_customer src ON tgt.customer_id = src.customer_id AND tgt.is_current = 1 WHERE ( tgt.name <> src.name OR tgt.email <> src.email OR tgt.address <> src.address ); -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ -- STEP 2: INSERT new version for changed customers -- (those just expired above) -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ INSERT INTO dim_customer_scd2 (customer_id, name, email, address, effective_date, expiry_date, is_current) SELECT src.customer_id, src.name, src.email, src.address, TRY_CAST(src.file_date AS DATE), -- effective_date = file_date '9999-12-31', -- open-ended 1 -- is_current = active FROM stg_customer src WHERE EXISTS ( -- customer existed but was just expired in step 1 SELECT 1 FROM dim_customer_scd2 tgt WHERE tgt.customer_id = src.customer_id AND tgt.is_current = 0 AND tgt.expiry_date = DATEADD(DAY, -1, TRY_CAST(src.file_date AS DATE)) ); -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ -- STEP 3: INSERT brand-new customers (never seen before) -- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ INSERT INTO dim_customer_scd2 (customer_id, name, email, address, effective_date, expiry_date, is_current) SELECT src.customer_id, src.name, src.email, src.address, TRY_CAST(src.file_date AS DATE), '9999-12-31', 1 FROM stg_customer src WHERE NOT EXISTS ( SELECT 1 FROM dim_customer_scd2 tgt WHERE tgt.customer_id = src.customer_id ); END;
⚡ PySpark Delta MERGE — Fabric Lakehouse Approach
🔥 When to use PySpark Delta MERGE vs T-SQL Stored Procedure
Use T-SQL SP when your target is a Fabric Warehouse (SQL-first, BI-ready, ACID T-SQL). Use PySpark Delta MERGE when your target is a Fabric Lakehouse Delta table (Spark-first, large-scale, part of a Medallion pipeline). Both achieve the same SCD2 result — the choice is about where your Gold layer lives.
Step 1 — Load CSV into Spark DataFrame and stamp file_date PySpark · Fabric Notebook
from pyspark.sql.functions import col, lit, to_date, current_date from delta.tables import DeltaTable # ── Load today's CSV (e.g. customer_2025-07-11.csv) file_date = "2025-07-11" # extracted from filename via ADF var_file_date stg_df = (spark.read .format("csv") .option("header", "true") .load(f"Files/customer_{file_date}.csv") .withColumn("file_date", lit(file_date)) .withColumn("effective_dt", to_date(lit(file_date), "yyyy-MM-dd")) ) stg_df.createOrReplaceTempView("stg_customer") stg_df.show()
Step 2 — Create dim_customer_scd2 Delta table (first run only) PySpark · Delta DDL
spark.sql(""" CREATE TABLE IF NOT EXISTS dim_customer_scd2 ( sk_customer BIGINT GENERATED ALWAYS AS IDENTITY, customer_id INT NOT NULL, name STRING, email STRING, address STRING, effective_date DATE NOT NULL, expiry_date DATE NOT NULL, is_current INT NOT NULL ) USING DELTA TBLPROPERTIES ('delta.enableChangeDataFeed' = 'true') """) # enableChangeDataFeed allows downstream consumers to # track exactly which rows were expired/inserted each run.
Step 3 — EXPIRE changed rows using Delta MERGE (whenMatched) EXPIRE STEP
# Build the "rows that changed" DataFrame changed_df = spark.sql(""" SELECT stg.* FROM stg_customer stg INNER JOIN dim_customer_scd2 dim ON stg.customer_id = dim.customer_id AND dim.is_current = 1 WHERE stg.name <> dim.name OR stg.email <> dim.email OR stg.address <> dim.address """) # EXPIRE: update is_current=0 and set expiry_date for changed rows dim_table = DeltaTable.forName(spark, "dim_customer_scd2") dim_table.alias("tgt").merge( changed_df.alias("src"), "tgt.customer_id = src.customer_id AND tgt.is_current = 1" ).whenMatchedUpdate(set={ "is_current" : lit(0), "expiry_date" : "date_sub(src.effective_dt, 1)" }).execute() print(f"Expired {changed_df.count()} changed rows")
Step 4 — INSERT new versions (changed + brand-new customers) INSERT STEP
# Union changed customers (new version) + brand-new customers new_versions_df = spark.sql(""" SELECT stg.customer_id, stg.name, stg.email, stg.address, stg.effective_dt AS effective_date, DATE'9999-12-31' AS expiry_date, 1 AS is_current FROM stg_customer stg WHERE -- new version for changed customers (expired above) EXISTS ( SELECT 1 FROM dim_customer_scd2 dim WHERE dim.customer_id = stg.customer_id AND dim.is_current = 0 AND dim.expiry_date = date_sub(stg.effective_dt, 1) ) OR -- brand-new customers never seen before NOT EXISTS ( SELECT 1 FROM dim_customer_scd2 dim WHERE dim.customer_id = stg.customer_id ) """) # Append new version rows — no MERGE needed, just insertOnly new_versions_df.write \ .format("delta") \ .mode("append") \ .saveAsTable("dim_customer_scd2") print(f"Inserted {new_versions_df.count()} new/updated rows")
Step 5 — Validate: current vs historical records Validation Queries
-- Current (active) customers — always 1 row per customer_id SELECT customer_id, name, address, effective_date FROM dim_customer_scd2 WHERE is_current = 1 ORDER BY customer_id; -- Full history — all versions per customer SELECT customer_id, name, address, effective_date, expiry_date, is_current FROM dim_customer_scd2 ORDER BY customer_id, effective_date; -- Point-in-time query: what did customer 2 look like on 2025-07-10? SELECT '2025-07-10' AS query_date, name, address FROM dim_customer_scd2 WHERE customer_id = 2 AND effective_date <= '2025-07-10' AND expiry_date >= '2025-07-10'; -- Returns: Jane Smith | California (her address BEFORE the change)
💡
Key Design Decisions
Why IDENTITY starts at 1001? Leaves room for reserved/unknown member rows at 0–1000. Unknown member rows (sk=0, name='Unknown') handle fact rows that arrive before the dimension is populated.

Why expiry_date = file_date - 1 day? Creates a non-overlapping effective/expiry range. Customer 2 was active California from Jul 10 → Jul 10. The new Los Angeles row is active from Jul 11 → 9999-12-31. This ensures clean point-in-time queries with simple BETWEEN logic.

Why 9999-12-31 instead of NULL for open-ended rows? NULL requires IS NULL checks everywhere. A sentinel date allows simple WHERE effective_date <= @query_date AND expiry_date >= @query_date without special-casing NULLs — critical for Power BI DAX measures.

Do NOT use DELETE in SCD2. Historical rows must never be deleted — that would permanently destroy audit history. Only UPDATE (to expire) and INSERT are valid operations in a Type 2 dimension.
